# 玩法范式 · 战斗单位多维养成（收集养成类）

> **用途**:`aigd-concept`/`aigd-system` 访谈"战斗单位养成"类系统(宠物/精灵/英雄/武将/可养成装备)时的领域弹药——一套被实战验证的结构骨架 + 设计要点 + 易错点。
> 本页是**抽象范式**,字段/表名是**示意**;落到具体项目时按该项目的命名走。
> 相关:数值设计风险见 [`../数值坑集/常见数值陷阱.md`](../数值坑集/常见数值陷阱.md);循环位置见 [`常见核心循环.md`](常见核心循环.md)「收集→养成→验证」。

## 一、范式骨架:一个"可被多条轴养成的战斗单位"

战斗单位养成 = **一个静态单位定义** + **N 条相互正交的成长轴** + **一个能力天花板总开关**。

```text
品质/稀有度(rarity) ── 总开关:决定 等级上限/进化次数/星级上限 ──┐
                                                              ▼
单位静态表(unit) ─┬─ 升级轴 (lv:   材料×量 → 三围%)
                  ├─ 进化轴 (换形态 id: 阶段 → 三围% + 解锁)
                  ├─ 升星轴 (star:  材料×量 → 三围% + 技能点)
                  ├─ 技能轴 (技能槽: 条件解锁 + 等级上限)
                  ├─ 编队轴 (编队位: 条件解锁)
                  └─ 羁绊轴 (成员组 + 阶梯条件 → 属性)
```

每条轴**独立配表、独立消耗、独立加成**,运行时各取当前档位行求和并入总属性。轴之间只通过"品质天花板"和"解锁条件"耦合,不互相内联——这是能并行迭代、能单独平衡的关键。

## 二、五个值得复用的设计决策

### 1. 收集单位 ≠ 形态单位(进化换 id)
进化后单位**换成不同 id**(`unit.evolveTo` = 下一形态 id)。于是需要一个**稳定的收集单位**贯穿全程:用一个进化全程不变的"线 id"(如 `unit.line`)做图鉴/羁绊归一。
- **真源唯一**:进化链以 `unit.evolveTo` 为唯一真源;"进化线成员表"(有序成员 `evolveLine.unit[]`)只作**收集/图鉴归一**,不当进化逻辑真源。同一关系存两处 → 必须指定谁是权威,否则双写漂移。
- **末阶哨兵**:`evolveTo = 0` 表示已末阶不可再进化。**id 恒正**,故 `0` 可安全做"无引用"哨兵(见陷阱页 §5,但要确认 id 域不含 0)。

### 2. 品质 = 能力天花板的总开关(硬封顶)
`rarityCap[品质]` 一行集中定义该品质的 `等级上限/可进化次数/星级上限`。**不管其他配置怎么填,运行时一律以品质封顶**(品质封顶优先)。
- 配合**补全数据策略**:进化/升星表含**超出该品质上限的模板行**(便于导表规整),运行时**恒以 `rarityCap[r].进化次数` 截断**,超出行不使用。
- 好处:数值表可整齐铺满,不必为每个品质裁剪;封顶逻辑集中一处。
- 风险:校验时"链长 > 品质次数"是**预期**而非错误 → 校验规则须用 `severity: advisory`(见 `checks/example.checks.json` 的 cardinality 规则)。

### 3. 累计值 vs 增量值(存总值,差分取单次)
三围加成 `*Percentage` 与技能点等存的是**截至该档位的累计总值**,不是单档增量。
- 结算:直接取当前档位行(如 `levelStep[当前等级].HpPercentage`),**不逐档累加**——防 N 档求和的精度/性能问题。
- 单次发放 = `本档总值 − 上一档总值`(如升 1 星新增技能点 = `starStep[star].skillPoint − starStep[star-1].skillPoint`)。
- 不变式:累计列**随档位单调不减**(`value_check` 的 `monotonic` 规则守这条)。

### 4. 多类单位复用同一张表
用一个标志位(如 `unit.isEnemy`)切换语义:敌方单位用 `hp/atk/def` **绝对值**;友方单位三围走百分比成长、绝对列留空。同一 schema 承载两类战斗单位,省一张表 + 共用战斗加载器。

### 5. 解锁条件参数化(condition + para)
技能槽/编队位用 `(condition, para)` 二元组表达解锁门槛:`condition` 是枚举(0无限制/1单位星级/2队伍等级/…),`para` 是阈值。加新解锁维度 = 加枚举值,不动表结构。

## 三、关键引用链(外键骨架,可移植)

```text
unit.rarity     → 枚举 Rarity          (能力天花板键)
unit.element    → 枚举 Element         (联合主键分量,若按元素分模板)
unit.line       → evolveLine.id        (收集单位)
unit.evolveTo   → unit.id (或 0=末阶)  (进化换 id 真源)
unit.skill[]    → 技能系统表           (跨系统[外部])
evolveStep/starStep (rarity, element) → 联合主键,经「复合键映射.json」分量匹配
bond.member[]   → unit.line            (按进化线归一,非按形态 id)
*.cost          → 道具/材料表.id       (各轴消耗)
*.property      → 属性表               (羁绊/战斗属性)
```

复合主键(如 `starStep:[rarity,element,star]`)要在配置目录登记 `复合键映射.json`,`value_check` 才能解析 `表[主键].字段` 引用。

## 四、设计访谈检查单(套这类系统时逐项问)

- [ ] **收集单位**是什么?进化/转职会不会换 id?换了的话靠什么贯穿(图鉴/羁绊)?
- [ ] **天花板总开关**:谁封顶能力(品质/稀有度/职业)?上限集中一处还是散落?
- [ ] **几条养成轴**?每条:消耗什么、产出什么、和别的轴是否正交?
- [ ] 加成是**累计值还是增量值**?(定一种,全表统一,写进配置说明)
- [ ] **补全数据**:数值表要不要铺超上限的模板行?运行时靠什么截断?
- [ ] **解锁条件**有几种维度?能否用 `(condition,para)` 参数化避免改结构?
- [ ] 同表是否复用于**多类单位**(敌方/友方/可召唤)?用什么字段切?
- [ ] 跨系统依赖(技能/科技/道具/文本)哪些是 `[外部]`?先标依赖、别内联。

## 五、本范式在循环中的位置

战斗单位养成是「**收集 → 养成 → 验证**」核心循环的"养成"中段:收集(抽卡/掉落)喂入新单位与材料,养成(本范式)把数值轴推高,验证(关卡/PVP)消费战力并回吐材料。设计时要确认**养成产出的战力提升,验证端有对应的消耗坡度**,否则养成失速(见 [`常见核心循环.md`](常见核心循环.md))。
